> 实现阶段引入的 bug 只能通过**后置验证**发现。

## 二、四方案优缺点对比

### 方案 A：中书省二次审核

**机制：** 执行部门完成后，方案提交回中书省做结果审查（检查代码、测试、文档）；中书省可批准或要求修改。

**新增状态机：**
```
executing → post_review → done（批准）
                        → revising → executing（要求修改）
```

| 维度 | 评估 |
|------|------|
| **成本** | 高：每次执行都多一轮中书省介入，token 消耗 +30-50% |
| **延迟** | 高：每个奏折额外 0.5-1 天（等中书省轮询+处理） |
| **覆盖率** | 中：中书省了解规划背景，但未必具备每个技术领域的深度（如 Python 细节、数据库 API 差异） |
| **实现复杂度** | 高：需新增 `post_review` 状态、修改 dispatcher、修改 chaoting CLI、修改 CAS 保护逻辑 |
| **主要优点** | 审查者了解原始计划意图，能判断实现是否"按计划来的" |
| **主要缺点** | 中书省变成瓶颈；技术审查深度有限；对小改动浪费严重 |

**适用场景：** 高价值、长周期任务，审查重点是"功能完整性"而非"代码质量"。

---

### 方案 B：对口部门审核

**机制：** 按任务类型分派给专业对口审核部门，例如：
- 兵部实现功能 → 刑部做安全审计
- 工部部署服务 → 兵部做代码审查（如有代码改动）
- 户部做数据迁移 → 由另一个户部实例或资深 agent 复验

**新增角色映射：**
```python
PEER_REVIEW_MAP = {
    "bingbu":  "xingbu",    # 兵部完成 → 刑部安全复查
    "gongbu":  "bingbu",    # 工部完成 → 兵部代码复查（如有代码改动）
    "hubu":    "xingbu",    # 户部完成 → 刑部数据安全复查
}
```

| 维度 | 评估 |
|------|------|
| **成本** | 中：只在有对口部门的场景下触发 |
| **延迟** | 中：额外等待对口部门响应（0.5 天左右） |
| **覆盖率** | 高：专业部门做专业审查，技术深度高于中书省 |
| **实现复杂度** | 高：需维护映射表；对口部门是否具备审查能力需额外设计 |
| **主要优点** | 专业对口，审查质量高；刑部做安全审计是天然的职责延伸 |
| **主要缺点** | 映射关系难以统一；部门职责边界模糊（刑部负责实施？还是审计？）；增加 agent 数量 |

**适用场景：** 安全敏感任务（数据处理、权限变更）效果最好。

---

### 方案 C：混合模式（按优先级/风险级别决定）

**机制：** 不对所有任务做二次审核，只对高风险奏折触发，与现有 `review_required` 四级标准联动：

| review_required | 执行后验收 | 验收方式 |
|-----------------|-----------|---------|
| 0（免审） | ❌ 不需要 | 执行完即完成 |
| 1（风险单审） | ❌ 不需要 | 执行完即完成 |
| 2（技术+风险双审） | ✅ 轻量验证 | 兵部必须在 Thread 发送含测试结果的反馈；司礼监确认 |
| 3（全审，军国大事） | ✅ 正式验收 | 中书省或指定给事中做二次确认 |

**实现方式：**
- review=0/1：现状不变，`chaoting done` 直接完成
- review=2：`chaoting done` 后强制要求 Thread 反馈包含测试日志（POLICY-thread-feedback.md 已有基础）
- review=3：`chaoting done` 后进入 `pending_acceptance` 状态，由中书省或司礼监显式确认

| 维度 | 评估 |
|------|------|
| **成本** | 低-中：只有 review=2/3 的奏折额外处理；大多数日常任务无影响 |
| **延迟** | 低-中：review=2 仅需 Thread 反馈确认（分钟级）；review=3 需人工确认（小时级） |
| **覆盖率** | 中-高：高风险任务全部覆盖，低风险任务跳过，符合投入产出比原则 |
| **实现复杂度** | 中：review=2 只需增强 POLICY 约束（不改代码）；review=3 需新增 `pending_acceptance` 状态 |
| **主要优点** | 与现有 review_required 体系无缝集成；低风险任务零额外开销；渐进式引入 |
| **主要缺点** | review=2 的"强制 Thread 反馈"实际上依赖 agent 自律，无系统强制；review=3 验收需占用人力 |

**适用场景：** 最适合当前朝廷系统阶段——既能覆盖高风险任务，又不让日常任务陷入官僚流程。

---

### 方案 D：随机抽查

**机制：** 所有奏折执行完后，N% 概率触发二次审查（由 dispatcher 决定，基于随机数或奏折 ID 哈希）。

```python
import hashlib
def should_sample(zouzhe_id, rate=0.2):
    # 确定性抽样：同一 ID 每次结果相同
    h = int(hashlib.md5(zouzhe_id.encode()).hexdigest(), 16)
    return (h % 100) < (rate * 100)
```

| 维度 | 评估 |
|------|------|
| **成本** | 低：只有 N% 任务有额外开销（如 N=20%，80% 任务不受影响） |
| **延迟** | 低：大多数任务无延迟；被抽到的延迟 0.5-1 天 |
| **覆盖率** | 低-中：随机覆盖，高风险任务不保证被抽到；样本量小时统计意义有限 |
| **实现复杂度** | 低：dispatcher 加随机逻辑即可，不需新状态 |
| **主要优点** | 实现最简单；有威慑效果（agent 知道可能被抽查）；开销可控 |
| **主要缺点** | 高风险任务可能恰好不被抽到；质量保障具有随机性，不稳定；难以向司礼监汇报"质量有保障" |

**适用场景：** 适合作为方案 C 的补充（对 review=1 的任务做随机抽样），不适合作为唯一质量机制。

## 四、推荐方案

### 主推：方案 C（混合模式）+ 增强现有 Thread 反馈规范

**理由：**

1. **问题根因是执行验证不足，不是审核缺失**  
   ZZ-003 和 ZZ-009 都是实现阶段的 API 用法错误，属于"测试没覆盖到"而非"审核没看到"。解决方案应是**强化执行部门的自测要求**，而非加一个新的审核角色。

2. **与现有 review_required 体系自然对齐**  
   方案 C 直接利用已有的四级标准，不引入新概念，学习成本为零。

3. **review=2/3 的任务已经过门下省双审，再加重量级后置审核成本过高**  
   对于 review=3（军国大事），加入 `pending_acceptance` 是合理的；对于 review=2，要求执行部门提供完整的测试日志即可。

4. **渐进实施，可分两步落地**

### 分步实施路径（仅建议，不涉及代码）

**第一步（即刻可行，无需改代码）：**
- 更新 POLICY-thread-feedback.md：要求 review=2 及以上任务的 Thread 反馈必须包含可复现的测试步骤和输出结果
- 更新执行部门 SOUL.md：明确验收标准检查清单（不只是"测试通过了"，而是列出测试命令和输出）

**第二步（v0.3 阶段，需少量代码）：**
- 为 review=3 的奏折新增 `pending_acceptance` 状态
- 执行部门调用 `chaoting done` 后自动进入此状态，等待司礼监或中书省显式调用 `chaoting accept` 确认
- 超时（如 24h）未确认则自动转为 `done`（防止阻塞）

**补充：方案 D 作为 review=1 的轻量保障**
- 对 review=1 任务实施 20% 随机抽样，被抽到的任务需补充测试报告
- 实现简单，无需新状态，dispatcher 加几行即可
- 威慑效果 > 实际覆盖率

## 五、豁免条件建议

以下情况建议**豁免二次审核**（即使 review=3）：

| 豁免条件 | 理由 |
|---------|------|
| `priority = critical`（紧急奏折） | 时间价值高于质量保障成本；紧急情况下先上线再补测 |
| 改动行数 < 100 行 | 小改动风险低，审查收益递减 |
| 纯文档改动（docs/ 目录，无代码变更） | 文档错误风险低，不影响运行时行为 |
| 仅测试用例改动（test_*.py 文件） | 测试本身就是验证手段，无需再测试测试 |
| 已有同类任务最近 7 天内通过二次审核 | 同类型任务的审核结果具有参考性，避免重复审查 |

**不得豁免的情况：**
- 涉及权限变更、密钥操作、生产数据变更
- 修改 dispatcher.py 核心逻辑
- 数据库 Schema 变更（DROP/ALTER）

## 六、讨论议题

以下问题供各部门发表意见：

**Q1（必要性）：** 目前 13% 的返工率是否可接受？还是应该通过二次审核降低？

> 吏部观点：13% 的返工率对应每 7-8 个任务就有 1 次返工，成本不低。但返工主要集中在实现质量（API 用法、单元测试覆盖），更应该通过**提高执行标准**而非增加审核轮次来解决。

**Q2（中书省负担）：** 如果中书省需要做二次审核，是否会成为瓶颈？

> 吏部观点：中书省同时负责规划和审核，加入二次审核会让其成为所有任务的必经节点，极易成为瓶颈。如需中书省参与，应限定在 review=3 的重大任务上。

**Q3（兵部意见）：** 被二次审核会显著增加任务周期感知吗？

> 吏部观点：对于兵部，更有价值的机制是**明确的测试验收清单**（acceptance_criteria 中列出测试命令），而不是等待另一个部门来审查。测试清单让兵部自测就能覆盖大多数问题。

**Q4（给事中角色延伸）：** 技术给事中是否应该在执行完成后再做一轮技术审查？

> 吏部观点：给事中的价值在于**审查计划的可行性**，而非**审查结果的正确性**。将给事中引入后置审核会混淆其职责定位，且给事中可能缺乏运行时验证能力。

## 七、结论

| 项目 | 建议 |
|------|------|
| **是否引入二次审核** | 是，但仅对 review=2/3 的任务，且方式轻量 |
| **推荐方案** | 方案 C（混合模式），配合方案 D（review=1 随机抽样 20%） |
| **第一步（无代码）** | 更新 POLICY：review=2+ 的 Thread 反馈必须包含可复现测试步骤 |
| **第二步（v0.3）** | 为 review=3 新增 `pending_acceptance` 状态 |
| **优先解决** | 在执行部门 SOUL.md 中加入更严格的自测要求（`acceptance_criteria` 必须包含测试命令） |
| **不建议** | 中书省全量二次审核（成本过高）、对口部门审核（职责混淆，实现复杂） |
