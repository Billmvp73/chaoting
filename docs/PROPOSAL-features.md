#### 1.1 奏折查询与检索

**ROADMAP 对应：** ✅ v0.3「CLI 观测命令（list/status/logs）」

**需求解析：** 支持按状态、优先级、时间范围筛选，基础能力 ROADMAP 已规划。建议在 v0.3 计划的基础上补充时间范围筛选参数。

**建议实现：**
```bash
chaoting list [--state created|planning|reviewing|executing|done|failed]
             [--priority low|normal|high|critical]
             [--agent bingbu|gongbu|...]
             [--since 2026-03-01]
             [--limit 20]

chaoting status <ZZ-ID>   # 详情 + 流转历史 + 投票记录
chaoting logs <ZZ-ID>     # zoubao 流水
```

| 项目 | 评估 |
|------|------|
| 工作量 | S（在 ROADMAP 规划基础上微调） |
| 建议阶段 | **v0.3**（与 ROADMAP 对齐） |
| 依赖 | 无 |

#### 1.2 批量奏折处理

**ROADMAP 对应：** 🆕 新增需求，ROADMAP 未提及

**需求解析：** 支持批量创建、批量状态变更（如批量标记超时任务为 failed）、批量导出。

**建议实现（分两步）：**
- **批量创建**：`chaoting batch-create <template.yaml>` 从 YAML 文件批量提交奏折
- **批量操作**：`chaoting batch-fail --state timeout --before 2026-01-01 "批量清理"` 管理员专用

**注意：** 批量操作风险较高，需配合 v0.3 的观测命令才能安全使用（先 list 再 batch）。

| 项目 | 评估 |

| 工作量 | M（批量创建 S，批量操作需安全防护 M） |

| 依赖 | 1.1 奏折查询（list 命令先行） |
| 投入产出比 | ⭐⭐⭐（有用但非紧迫） |

#### 1.3 审核流程自动化

**ROADMAP 对应：** ✅ 部分对应 v0.4「Flight Rules 引擎」

**需求解析：** 原始需求包含"触发条件、自动分配"。当前审核人在奏折创建时静态指定，自动化意味着基于规则动态决定是否审核及审核人选。

**建议实现（两个子功能）：**

**子功能 A：审核规则引擎**（新增）
基于奏折属性（优先级、执行部门、关键词）自动设置 `review_required` 级别和 `review_agents`：
```python
# 规则示例（存于配置文件）
rules = [
    {"if": {"priority": "critical"}, "then": {"review_required": 3}},
    {"if": {"target_agent": "xingbu"}, "then": {"review_required": 3}},
    {"if": {"title_contains": "数据迁移"}, "then": {"review_required": 2}},
]
```

**子功能 B：Flight Rules 引擎**（已在 ROADMAP v0.4）
历史教训自动注入给事中审核上下文，辅助判断。

| 项目 | 评估 |
|------|------|
| 工作量（规则引擎）| M |
| 工作量（Flight Rules）| M（已在 ROADMAP） |
| 建议阶段 | **v0.4**（规则引擎与 Flight Rules 同期） |
| 依赖 | review_required 四级标准需先确定（见 PROPOSAL-governance.md） |
| 投入产出比 | ⭐⭐⭐⭐（减少人工配置审核级别的负担） |

#### 1.4 奏折模板库

**ROADMAP 对应：** ✅ v1.0「任务模板」，当前规划为最低优先级

**需求解析：** 预定义常用任务结构（Bug修复、新功能、数据迁移等），降低奏折创建门槛。

**建议调整：** 将优先级从 v1.0 **提前到 v0.4**。理由：

1. 模板本质上是 YAML/JSON 文件，实现简单（S 工作量）
2. 模板可极大提升中书省规划质量（标准化上下文）
3. 是批量处理（1.2）的前置依赖——批量创建需要模板

**建议实现：**
```bash
chaoting new --template bug-fix "标题" "描述"
chaoting templates list    # 列出可用模板
```

模板文件格式：
```yaml
# templates/bug-fix.yaml
name: bug-fix
description: 修复 Bug 模板
review_required: 1
suggested_agent: bingbu
description_template: |
  ## 问题描述
  {description}
  ## 复现步骤
  1. ...
  ## 预期行为
  ...
  ## 验收标准
  修复后该路径不再出现此错误
```

| 项目 | 评估 |
|------|------|
| 工作量 | S |
| 建议阶段 | **v0.4**（从 v1.0 提前） |
| 依赖 | 无（但会成为批量处理的前置依赖） |
| 投入产出比 | ⭐⭐⭐⭐（低成本高收益，提升一致性） |

### 第三优先级——运维与扩展

#### 3.1 数据备份与恢复机制

**ROADMAP 对应：** 🆕 新增需求

**需求解析：** SQLite 文件定期备份，支持从备份恢复。

**建议实现（轻量方案）：**
```bash
# 定期备份脚本（加入 cron）
chaoting backup [--output /path/to/backup/]
# → 执行 sqlite3 .backup，输出带时间戳的 .db 文件

# 恢复
chaoting restore /path/to/chaoting.db.2026-03-08.bak
```

**简单方案：** SQLite 的 `.backup` 命令天然支持热备份（不影响并发读写）。整个功能实现极简。

| 项目 | 评估 |
|------|------|
| 工作量 | S |
| 建议阶段 | **v0.3**（生产环境必需，工作量小） |
| 依赖 | 无 |
| 投入产出比 | ⭐⭐⭐⭐⭐（数据安全兜底，实现极简） |

#### 3.3 多数据库支持（PostgreSQL 等）

**ROADMAP 对应：** 🚫 ROADMAP 明确不做

**评估：** 维持 ROADMAP 决策，**不纳入规划**。

理由：
1. SQLite WAL 模式已支持并发读写，满足当前单节点场景
2. 迁移成本高（需抽象 DB 层），收益不明确
3. 如未来确需扩展，应通过 ORM 层（SQLAlchemy）统一抽象，而非直接支持多 DB

**建议后续观察：** 如果系统规模扩大到多节点部署需求，届时重新评估。

## 六、关键建议总结

1. **立即启动 v0.3 的数据备份功能**：生产环境没有备份是最大的运维风险，工作量极小（S）。

2. **将奏折模板库从 v1.0 提前到 v0.4**：实现简单（S），但对提升中书省规划质量和批量处理支持的价值巨大。

4. **Web Dashboard 不建议提前**：L 级工作量投入大，命令行仪表盘（S）可先满足 80% 的观测需求。

5. **多数据库、权限细化、插件化维持 Backlog**：ROADMAP 决策合理，当前阶段不应分散精力。

6. **Discord 集成分两步**：通知推送（S，v0.3）先行，命令交互（M，v1.0）延后，避免过早复杂化。
