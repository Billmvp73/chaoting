## v0.3 — 可用性增强

> **目标：** 让系统从"能跑"变成"好用"。补齐日常运维必需的观测和容错能力。

### 1. 用户推送通知

**是什么：** 任务完成、失败、超时时，通过 OpenClaw 消息通道（Signal/Telegram/Discord）主动推送给用户，而不仅写入数据库。

**为什么需要：** 目前用户必须主动查询才能知道任务状态，无感知。任务失败或长时间未响应时用户毫无察觉，严重影响使用体验。司礼监已有 `notify_capcom` 基础，需扩展到普通完成/失败事件。

**实现要点：**
- `chaoting done` / `chaoting fail` 时触发 OpenClaw `message` 工具通知
- 通知内容：奏折 ID、标题、状态、产出摘要/失败原因
- 支持配置：可选开关、目标渠道

**预估工作量：** S（改动集中在 `dispatcher.py` 和 `chaoting` CLI）

---

### 2. CLI 观测命令（list / status / logs）

**是什么：** 新增三条 CLI 命令：
- `chaoting list [--state <状态>] [--agent <角色>] [--limit N]`：列出奏折
- `chaoting status <id>`：查看单个奏折的完整状态（含流转历史、投票记录）
- `chaoting logs <id>`：查看奏折的奏报（zoubao）流水

**为什么需要：** 当前无法从命令行快速查看系统状态。调试、运维、演示时只能直接查 SQLite，门槛高且容易误操作。

**实现要点：**
- 纯只读 SQL 查询，格式化输出
- `list` 支持按状态/部门/优先级过滤，默认显示最近 20 条
- `status` 一屏展示所有关键信息

**预估工作量：** S

---

### 3. Agent 故障转移（Fallback）

**是什么：** 当目标 Agent 连续超时达到 `max_retries` 上限时，自动将任务转派给同类备用 Agent（如 `bingbu` 主挂 → 转 `bingbu_2`），而非直接进入 `timeout` 状态。

**为什么需要：** 当前重试耗尽即宣告失败，但失败原因往往是 Agent 临时不可用而非任务本身有问题。Fallback 可大幅提升系统韧性，避免无谓的人工干预。

**实现要点：**
- 在 `zouzhe` 表或全局配置中定义 Fallback 映射（`bingbu → [bingbu_2, gongbu]`）
- Dispatcher 在最后一次重试超时后查询 Fallback 列表，转派并记录 `liuzhuan`
- 无 Fallback 可用时才进入 `timeout`

**预估工作量：** M

## v0.4 — 智能化

> **目标：** 让系统从"被动执行"变成"主动辅助"。引入知识积累和任务编排能力。

### 4. 任务依赖（前置条件）

**是什么：** 支持在创建奏折时声明前置任务（`depends_on: ["ZZ-XXXXXXXX-001"]`），调度器在所有前置任务完成前不派发当前任务。

**为什么需要：** 当前所有任务相互独立，复杂项目（如：先建数据库 Schema → 再写业务逻辑 → 最后写文档）需要人工控制顺序，容易出错或并发冲突。

**实现要点：**
- `zouzhe` 表新增 `depends_on TEXT`（JSON array of ZZ IDs）
- Dispatcher 在派发前检查所有依赖是否为 `done` 状态
- `chaoting list` 展示依赖关系图

**预估工作量：** M

---

### 5. Flight Rules 引擎（基于前车之鉴）

**是什么：** 在 `qianche`（前车之鉴）表基础上，构建自动建议引擎：当新奏折与历史失败/重试任务相似时，在 `pull` 返回结果中高亮相关经验教训，辅助 Agent 规避已知风险。

**为什么需要：** `qianche` 表已存在但未被主动利用——Agent 只有在 `pull` 时看到全部记录，无法快速识别相关内容。Flight Rules 将历史教训转化为可操作的决策辅助，降低重蹈覆辙的概率。

**实现要点：**
- `chaoting pull` 时按 `agent_role` + 关键词匹配 `qianche`，返回 top-3 相关记录
- 支持 `chaoting lesson <id> <text>` 快捷命令写入 `qianche`（简化当前 `context` 命令）
- 给事中审核时自动注入相关 `qianche` 记录，辅助判断

**预估工作量：** M

---

### 6. 典籍（dianji）自动注入与过期清理

**是什么：** 执行部门完成任务时，自动从产出内容中提取关键上下文写入 `dianji`；同时为 `dianji` 增加 TTL（存活时间），定期将过期条目标记为 `stale`。

**为什么需要：** 当前典籍完全依赖 Agent 手动调用 `chaoting context` 写入，实际使用率低。自动提取可让知识库随任务自然增长；TTL 清理防止过时信息误导后续 Agent。

**实现要点：**
- `chaoting done` 时可附带 `--context key=value` 批量写入 `dianji`
- Dispatcher 定期扫描 `dianji`，将超过 30 天未更新且 `confidence=fresh` 的记录降级为 `stale`
- `dianji` 表新增 `expires_at` 字段

**预估工作量：** S

## v1.0 — 平台化

> **目标：** 让系统从"命令行工具"变成"可观测平台"。提供可视化界面和标准化任务模板。

### 7. Web Dashboard

**是什么：** 基于 Web 的任务看板，展示所有奏折的实时状态、流转历史、给事中投票情况，支持手动触发重试、撤销任务等操作。

**为什么需要：** 命令行界面对非技术用户不友好，且难以直观呈现多任务并发状态。Dashboard 是系统走向真实生产环境的必要门槛。

**实现要点：**
- 轻量 Web UI（推荐：FastAPI 后端 + HTMX 前端，避免引入重型 React 构建链）
- 任务列表（可过滤/排序）、详情页、流转时间线、投票面板
- 只读优先，写操作（重试/撤销）需二次确认

**预估工作量：** L

---

### 8. 任务模板（Template）

**是什么：** 预定义常用任务的描述和规划结构，用户可通过模板快速创建标准化奏折（如"修复 Bug 模板"、"新增 API 端点模板"、"数据迁移模板"）。

**为什么需要：** 当前每次创建任务都从头描述，重复劳动多且质量不稳定。模板降低任务创建门槛，同时为中书省规划提供更清晰的上下文。

**实现要点：**
- YAML/JSON 模板文件存于 `<CHAOTING_DIR>/templates/`
- `chaoting new --template bug-fix "标题" "描述"` 快速创建
- 模板包含：描述框架、默认 `review_required` 级别、推荐执行部门

**预估工作量：** S

---

### 9. 运营指标统计

**是什么：** 定期生成系统运营报告：任务完成率、平均耗时、各部门负载、封驳率、超时率等核心指标。

**为什么需要：** 无指标则无优化方向。统计数据可帮助识别瓶颈（哪个部门最慢？哪类任务封驳率最高？）和调优参数（超时时间是否合理？重试次数够不够？）。

**实现要点：**
- `chaoting stats [--days 7]`：命令行输出指标摘要
- 指标来源：纯 SQL 聚合查询，无需额外存储
- v1.0 Dashboard 集成指标图表

**预估工作量：** S

## 刻意不做的事

以下功能经过评估后**暂不纳入规划**，原因如下：

| 功能 | 原因 |
|------|------|
| 多租户 / 权限系统 | 当前用户群体为单人或小团队，引入权限管理增加复杂度但收益有限 |
| 投票权重 / 加权决策 | 当前一票否决机制已足够，权重系统增加配置复杂度 |
| 分布式部署 / 多节点 | SQLite WAL 已满足当前规模，分布式引入网络分区等新问题 |
| 插件系统 | 过早抽象，等有第三方接入需求时再考虑 |
